## โปรเจกต์
- **MEE Stock** (เดิม ElecStock) — ระบบบริหารสต๊อกวัสดุไฟฟ้างานก่อสร้าง
- Stack: Next.js 14 (App Router), Express, PostgreSQL/Prisma, TypeScript
- Deploy: Vercel (frontend) + Railway (backend/DB)
- Tracker 33 วัน MVP→Executive Dashboard: รายละเอียดใน `docs/requirements.md`, `docs/changelog.md`
- Dev env: WSL2/Ubuntu + Claude Code

## สถาปัตยกรรม Finalized
1. **Ledger-based balance** — ยอดคงเหลือคำนวณจาก transaction เสมอ ไม่ mutate ตรง (auditability, กัน race condition)
2. **Multi-stage material issue workflow** — ต้องอนุมัติหลายขั้นก่อนจ่ายจริง
3. **2D access control** — แยก Role (สิทธิ์การกระทำ) ออกจาก Data Visibility (ขอบเขตไซต์ที่เห็น) เพื่อรองรับ role เดียวกันเห็นข้อมูลต่างไซต์

Executive dashboard: stock value + trend 6 เดือน + top 5 วัสดุเบิกมากสุด + top 5 ไซต์ต้นทุนสูงสุด + ต่ำกว่า reorder point, gate `role EXECUTIVE` + `accessLevel MANAGER|ADMIN`

## แก้บั๊กความปลอดภัยร้ายแรง (2026-07-07) — passwordHash รั่วผ่าน nested employee relations

ระหว่างเตรียม backend สำหรับ Material Issue UI (เพิ่ม `/warehouses`) เจอว่า `passwordHash` (bcrypt hash) รั่วออกไปจริงผ่าน **4 endpoints**: `GET /material-issues*` (requester/fulfilledBy), `GET /purchase-orders*` (createdBy), `GET /goods-receives*` (createdBy), `GET /stock/ledger` (performedBy) — ทุก authenticated user เรียกได้หมด ไม่ต้อง role พิเศษ. สาเหตุ: repository เหล่านี้ include ความสัมพันธ์ Employee ด้วย `fieldName: true` ตรงๆ (ดึงมาทุก column รวม `passwordHash`) แทนที่จะใช้ `select` แบบที่ `employeeRepository.findEmployeeById` ทำอยู่แล้ว (คอมเมนต์ในไฟล์นั้นเขียนไว้ชัดว่า "Excludes passwordHash" แต่ pattern นี้ไม่ได้ถูกใช้ซ้ำที่อื่นเลย).

**แก้แล้ว**: เพิ่ม `employeeRefSelect` (export จาก `employeeRepository.ts`, มีแค่ id/name/email/role/accessLevel) แล้วเปลี่ยนทั้ง 4 จุดจาก `relationName: true` เป็น `relationName: { select: employeeRefSelect }`. Verified ด้วย curl ว่า `passwordHash` หายไปจากทุก endpoint ที่เจอปัญหา, ยังคง name/role/accessLevel ไว้ให้ UI แสดงผลได้ปกติ. `pnpm run test` ผ่านครบ 20/20 เหมือนเดิม (ไม่มี test ที่ assert ตรงๆ กับ field ที่ตัดออก). Comment ไว้ใน `employeeRefSelect` ว่าให้ใช้ pattern นี้ทุกครั้งที่มี Employee relation ใหม่ในอนาคต กันไม่ให้เกิดซ้ำ.

- **known limitation ของ curl-based verification (ไม่ใช่บั๊กจริง)**: Server Action ที่เป็น **plain/unbound** (เช่น `createMaterialAction`, `createMaterialIssueAction` — รับ `FormData` ตรงๆ ไม่ใช่ `.bind(null,id)`) จำลองผ่าน curl ไม่ได้ เจอ "Failed to find Server Action" แม้ manifest ถูกต้อง — ไล่เข้าไปใน Next 14.2.35 compiled runtime แล้วสรุปว่าเป็น edge case เฉพาะตอนจำลอง no-JS fallback ผ่าน curl เท่านั้น ไม่ใช่บั๊กที่ browser จริง (เปิด JS) จะเจอ เพราะ browser ใช้ resolution path คนละแบบ (header-based). Action ที่เป็น **bound** (`.bind(null,id)`) ยืนยันผ่าน curl ได้จริงซ้ำหลายรอบ (deactivate, approve, fulfill) — **แนวทางที่ยึดต่อไป: material-issue actions ทั้งหมดออกแบบให้เป็น bound action ตั้งแต่แรกเพื่อให้ verify ผ่าน curl ได้โดยไม่ต้องพึ่ง browser จริง** ส่วน create-flow (ที่จำเป็นต้อง unbound เพราะรับ FormData เต็ม) verify ด้วยการ replicate logic ตรงไปที่ backend ผ่าน script แทน (ดู `PLAN.md` สำหรับรายละเอียดการไล่ debug เต็ม)
- **ผลข้างเคียงที่เจอระหว่างไล่บั๊กรอบก่อน (แก้แล้ว)**: curl control-test ทำให้ MAT-001 (seed material จริง) ถูก deactivate โดยไม่ตั้งใจ — คืนค่า `isActive:true` แล้วด้วย one-off script

### Material Issue workflow UI (เสร็จ 2026-07-07)
- **Design ที่ตกลงกับ user ก่อนทำ**: (1) เลือก warehouse ตัวเดียว, auto-derive project ให้จาก warehouse ที่เลือก — derive ที่ **server action** โดย fetch `/warehouses` หา `projectId` เอง ไม่รับ projectId จาก client เลย (กัน spoof). (2) Approve one-click เต็มจำนวนที่ขอ ไม่มี per-item override ใน UI (ทำผ่าน API ตรงได้อยู่แล้ว ถ้าจะทำเป็น fast-follow). (3) รองรับสแกนบาร์โค้ด — user ยังไม่มีบาร์โค้ดจริง ใช้ `Material.code` เป็นค่าสแกนไปก่อน (ไม่ migrate schema), แยก exact-match endpoint ต่างหากจาก fuzzy search กันไว้เผื่ออนาคตมีบาร์โค้ดผู้ผลิตจริงจะได้ไม่กระทบ UI
- **Backend เพิ่มก่อนเริ่ม UI**: `GET /warehouses` (site-visibility filtered, include `project`), `GET /materials/by-code/:code` (exact match สำหรับ barcode scan)
- **หน้า**: `/material-issues` (list + tabs: ทั้งหมด/รออนุมัติ/อนุมัติแล้ว/จ่ายบางส่วน/จ่ายครบแล้ว/ปฏิเสธ), `/material-issues/new` (warehouse picker + `ItemsField` client component: ช่องสแกนบาร์โค้ด auto-focus → `/api/materials/by-code/:code` proxy route ทุก Enter, บวกจำนวนอัตโนมัติถ้าสแกนซ้ำ, dropdown มือเป็น fallback), `/material-issues/[id]` (item breakdown + ผลอนุมัติ + ผู้จ่าย + ปุ่ม action role/status-aware: Approve/Reject เฉพาะ PENDING_APPROVAL + APPROVER/ADMIN + **ไม่ใช่ requester ตัวเอง** (ซ่อนปุ่มเลย ตรงกับ `FORBIDDEN_SELF_APPROVAL`), Fulfill เฉพาะ APPROVED + WAREHOUSE/ADMIN)
- **Verified จริงครบ flow**: create (verify logic ผ่าน script replicate ตรงไปที่ backend — ดูหมายเหตุ unbound-action ด้านบน), approve (bound action ผ่าน UI จริง, PENDING_APPROVAL→APPROVED), fulfill (bound action ผ่าน UI จริง, APPROVED→FULFILLED, สร้าง StockTransaction ISSUE จริง), self-approval hidden ถูกต้อง. typecheck + `pnpm run test` (20/20) + `next build` ผ่านหมด

ปิด TODO 2 ข้อสุดท้ายที่ต้องใช้ browser จริง (unbound create-forms × 4, print-preview CSS × 3 หน้า) โดยติดตั้ง Playwright + Chromium เป็น devDependency ใน `apps/web` (environment นี้ไม่มี root ให้รัน `playwright install-deps` เลย download shared libs ที่ขาด — `libnspr4`/`libnss3`/`libasound2` — ด้วย `apt-get download` (ไม่ต้อง root) + `dpkg-deb -x` แตกเข้าโฟลเดอร์แยก แล้วโหลดผ่าน `LD_LIBRARY_PATH`)
- **เจอบั๊กจริงระหว่างทดสอบ**: `createMaterialIssueAction` ผ่าน browser จริงได้ "Internal server error" ทุกครั้ง — สืบไปเจอว่า `generateDocNo()` (โค้ดซ้ำกันเป๊ะใน 3 ที่: `materialIssueService.ts`/`purchaseOrderService.ts`/`goodsReceiveService.ts`) ใช้ `COUNT(*)+1` แทนที่จะเป็น `MAX(suffix)+1` — เช็ค DB จริงพบว่ามีแค่ 10 rows แต่เลขสูงสุดที่ใช้ไปแล้วคือ 11 (เพราะมี gap ที่ 0009 จาก concurrent request ก่อนหน้า) ทำให้ COUNT+1 คำนวณได้ 11 ซ้ำกับ docNo ที่มีอยู่แล้ว**ตลอดไป** ไม่ใช่แค่ครั้งเดียว เพราะ insert ที่ fail ไม่ทำให้ count เปลี่ยน — retry เท่าไหร่ก็ชนซ้ำเดิม
- **แก้ที่ root cause**: เปลี่ยนทั้ง 3 service จาก count-based เป็น query หา docNo ล่าสุดที่ตรง prefix วันที่แล้ว extract ตัวเลขบวก 1 (`findLatestDocNoWithPrefix` เพิ่มใหม่ใน repository ทั้ง 3 ไฟล์ + `lib/docNoSequence.ts#extractDocNoSuffix`) — self-healing ไม่ว่าจะมี gap ประวัติศาสตร์แบบไหนก็ตาม. เพิ่ม `lib/docNoRetry.ts#createWithDocNoRetry` ครอบการ create ทั้ง 3 ที่ไว้ด้วย เผื่อ race จริงระหว่าง 2 request ที่อ่าน MAX พร้อมกันเป๊ะ (เคสที่ MAX-based fix เองยังไม่ครอบคลุม 100%)
- **เจอ infra outage กลางทางด้วย** (ไม่เกี่ยวกับโค้ด): Docker Desktop บน Windows host หยุดทำงาน (พา container `meestock-db` ดับไปด้วย), พร้อมๆ กับ sandbox's `/tmp` ถูกล้างและ background dev server ตายไปด้วย — วินิจฉัยผ่าน `docker.exe ps`/pipe connection error, แก้ได้หลัง user restart Docker Desktop + start container `meestock-db` เอง

- Playwright ถูกเก็บไว้เป็น devDependency ต่อ (ไม่ถอนออก) เผื่อใช้ verify รอบหน้า
