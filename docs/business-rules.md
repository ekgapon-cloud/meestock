# Business Rules — MEE Stock

## 1. กฎเรื่องยอดคงเหลือสต๊อก (Stock Balance Rules)

1. ยอดคงเหลือ (Stock Balance) **ต้องคำนวณจาก ledger transaction เท่านั้น** ห้าม update field ยอดคงเหลือโดยตรง
2. ทุก transaction ที่กระทบสต๊อกต้องมี type ชัดเจน เช่น `RECEIVE`, `ISSUE`, `ADJUSTMENT`, `RETURN`, `TRANSFER`
3. ห้ามลบ transaction ที่บันทึกไปแล้ว หากบันทึกผิดต้องสร้าง transaction ประเภท `ADJUSTMENT` เพื่อแก้ไข (reversal) พร้อมระบุเหตุผล
4. ยอดคงเหลือต้องไม่ติดลบ เว้นแต่จะเปิด flag เฉพาะกรณี (เช่น backorder) ซึ่งต้องมีการอนุมัติพิเศษ
5. **จำนวนวัสดุต้องเป็นจำนวนเต็ม (integer) เสมอ** ในทุกฟังก์ชัน — สั่งซื้อ (orderedQty), เบิก (requestedQty/approvedQty/issuedQty), รับของ (quantity), โอนย้าย (quantity), นับสต๊อก (actualQty), ปรับยอด (quantityChange). ห้ามกรอกทศนิยม. **ยกเว้นจำนวนเงิน** (unitCost, standardCost, contractValue, งบประมาณ) ที่กรอกทศนิยมได้ บังคับ 2 ชั้น: zod `.int()` ฝั่ง API + `step="1"` (และบล็อกปุ่ม `.`/`,`) ฝั่ง UI

## 2. กฎเรื่อง Material Issue Workflow (Multi-stage)

1. สถานะคำขอเบิกต้องเปลี่ยนตามลำดับ: `REQUESTED` → `APPROVED` → `ISSUED` (หรือ `REJECTED` ได้ในขั้นตอน approve)
2. ห้าม skip สถานะ เช่น เปลี่ยนจาก `REQUESTED` เป็น `ISSUED` โดยไม่ผ่าน `APPROVED`
3. ผู้ที่ขอเบิก (Requester) ห้ามเป็นผู้อนุมัติคำขอของตัวเอง (segregation of duties)
4. การจ่ายวัสดุจริง (`ISSUED`) ต้องสร้าง ledger transaction ประเภท `ISSUE` ทันทีในธุรกรรมเดียวกัน (atomic)
5. หากคำขอถูก `REJECTED` ต้องระบุเหตุผลเสมอ

## 3. กฎเรื่องสิทธิ์การเข้าถึงข้อมูล (Two-dimensional Access Control)

1. **มิติที่ 1 — Role**: กำหนดว่าผู้ใช้ "ทำอะไรได้บ้าง" (เช่น อนุมัติได้/เบิกได้/ดูรายงานได้)
2. **มิติที่ 2 — Data Visibility**: กำหนดว่าผู้ใช้ "เห็นข้อมูลของไซต์งานใดบ้าง"
3. ทุก query ที่ดึงข้อมูลระดับ record (เช่น รายการเบิก, ยอดคงเหลือ) ต้อง enforce ทั้งสองมิติพร้อมกันเสมอ ไม่สามารถอาศัย role อย่างเดียวในการกรองข้อมูลได้
4. Admin เท่านั้นที่สามารถเห็นข้อมูลทุกไซต์งานโดยไม่มีข้อจำกัด Data Visibility

## 4. กฎเรื่องการรับเข้าสต๊อก (Receiving)

1. การรับเข้าสต๊อกควรอ้างอิงใบสั่งซื้อ (PO) เสมอเมื่อเป็นไปได้
2. หากรับเข้าโดยไม่มี PO อ้างอิง (เช่น รับบริจาค/โอนจากไซต์อื่น) ต้องระบุประเภท transaction ที่ชัดเจน (เช่น `TRANSFER`, `DONATION`)

## 5. กฎเรื่องการปรับปรุงสต๊อก (Stock Adjustment)

1. การปรับปรุงยอดสต๊อก (เช่น จากการนับสต๊อกจริง) ต้องมีการอนุมัติจาก Storekeeper หรือ Admin เท่านั้น
2. ต้องบันทึกเหตุผลของการปรับปรุงทุกครั้ง

## 6. กฎด้านการเงิน/ต้นทุน (เชื่อมกับ site-costing)

1. มูลค่าวัสดุคงเหลือคำนวณจากต้นทุนเฉลี่ยถ่วงน้ำหนัก (Weighted Average Cost) เว้นแต่ระบุเป็นอย่างอื่น
2. ต้นทุนของแต่ละไซต์งานต้องแยกคำนวณจากการเบิกจ่ายจริงของไซต์นั้นเท่านั้น

> เอกสารนี้เป็น living document ควรอัปเดตทุกครั้งที่มีการเปลี่ยนแปลง business logic ที่กระทบ stock balance หรือ workflow
