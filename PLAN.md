## Refinement: bound-action curl verification

Confirmed while building Access Control that the deciding factor for curl-verifiability isn't "does the action take extra FormData" — `assignSiteAccessAction(id, formData)` bound via `.bind(null, user.id)` still takes a real `warehouseId` field from the form, and resolves fine via curl. The only thing that matters is whether `.bind()` was used *at all*. So the practical rule going forward: bind at least the id on any action that operates on an existing record, even if the form still has other fields to fill in; the only irreducible unbound case is a brand-new-record create form with no id yet to bind.

## Real-browser verification + docNo race-condition bugfix (2026-07-07)

Closed out the last two "needs a real browser" TODO items by installing Playwright + Chromium as a devDependency in `apps/web` (root access wasn't available for `playwright install-deps`, so the missing shared libs — `libnspr4`/`libnss3`/`libasound2` — were fetched via root-less `apt-get download` + `dpkg-deb -x` into a local dir and loaded via `LD_LIBRARY_PATH`). Scripted all 8 remaining checks: the 4 unbound create-forms (material/material-issue/PO/user) and print-preview CSS on the 3 document detail pages.

This surfaced a real, reproducible bug: `generateDocNo()` (duplicated identically in `materialIssueService.ts`/`purchaseOrderService.ts`/`goodsReceiveService.ts`) computed the next doc number as `COUNT(*)+1` over today's rows. Once any gap ever opened in the sequence (a failed create, a race between two overlapping requests), the count permanently undercounted relative to the highest suffix actually in use — so every subsequent create computed the same already-taken docNo and failed with a 500, forever, not just once. Root-caused via a live 500 hit during the Playwright run, confirmed by querying the dev DB directly (10 rows but highest suffix in use was 11, due to a gap at 0009 from earlier concurrent seeding). Fixed by replacing the count query with `findLatestDocNoWithPrefix` (new repository function in all 3 repos, `docNo` prefix match + `orderBy: desc`) and `extractDocNoSuffix` (new `lib/docNoSequence.ts`) — self-healing regardless of any historical gap. Also added `lib/docNoRetry.ts#createWithDocNoRetry` around each create call to retry on a genuine P2002 docNo collision from two truly concurrent requests (the MAX-based fix removes the permanent-collision case, but a race between two simultaneous reads of the same MAX is still theoretically possible).

Also hit, mid-session, a real infra outage unrelated to the code: Docker Desktop on the Windows host had stopped (taking the `meestock-db` Postgres container with it), which also coincided with the sandbox's `/tmp` being wiped and background dev-server processes dying. Diagnosed via `docker.exe ps`/pipe-connection errors; resolved once the user restarted Docker Desktop and the `meestock-db` container.

All 8 Playwright checks pass; typecheck + Vitest (20/20) pass after the docNo fix. One test-script false-positive along the way: the goods-receive print-CSS check expected a `.status-badge` that doesn't exist there by design (`GoodsReceive` has no lifecycle/status, unlike Material Issue/PO) — fixed the test assumption, not the product.

## Resolved: the "Failed to find Server Action" investigation

The `/materials/new` category-dropdown change itself is correct and verified (dropdown renders real categories, typecheck/build clean). Verifying the *form submission* via raw curl hit "Failed to find Server Action" consistently — reproduced in both dev **and** a clean production `next start`, ruling out the dev-hot-reload-cache theory from earlier. Traced into Next.js 14.2.35's own compiled runtime:

- Real browsers with JS enabled submit Server Actions as a "fetch action" — a `Next-Action` HTTP header plus a React-encoded body — resolved via a simple manifest lookup.
- The **no-JS progressive-enhancement fallback** (a plain `<form>` POST, which is what raw curl with `-F` flags simulates) is resolved differently: Next parses the multipart body looking for a `$ACTION_ID_<40-char-hash>` field and validates the hash against the module map itself.
- The **bound**-action wire format (`deactivateMaterialAction.bind(null, id)`, used on `/materials`) reliably works via curl in this same environment — repeatedly confirmed, including real DB side effects.
- The **plain/unbound**-action wire format (`createMaterialAction`, used on `/materials/new`) consistently fails resolution via the same raw-curl approach, even against a fresh production build whose `server-reference-manifest.json` correctly lists the action's hash.
- Adding a `Next-Action` header changes the error (from "Failed to find Server Action" to "Connection closed"), confirming the header-based path *does* find the action — the failure is specific to the body-field-based lookup for this one action, not the action itself being broken or missing from the build.

**Conclusion**: this is very likely a real edge case in Next 14.2.35's no-JS-fallback resolution path, only reachable by hand-simulating a raw multipart POST — not something an actual browser user (JS enabled, the standard case) would ever hit, since browsers use the header-based fetch-action path. Continuing to reverse-engineer the exact minified condition had poor ROI. **This needs a real browser (or Playwright) verification pass**, not curl, to close out with full confidence — no headless browser was available in this environment to do that here.

Side effects from this investigation, already cleaned up: an errant control-test curl call briefly deactivated the real seed material `MAT-001` — restored via a one-off script. All other test materials created during debugging (`MAT-TEST-01`, `MAT-CONTROL`) were left soft-deleted (inactive), which is harmless/expected soft-delete behavior, not corruption.
